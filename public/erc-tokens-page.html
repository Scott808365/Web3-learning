1. What Does EIP Mean?

üìò Definition:

EIP stands for Ethereum Improvement Proposal. It is a formal document that outlines a technical standard or feature suggestion for the Ethereum network. Anyone in the Ethereum community can create an EIP to propose:

  • A protocol change
  • A smart contract standard
  • A network upgrade
  • New governance or API methods

Think of it like a ‚Äúblueprint‚Äù for improvements or new functionalities.

üß± Types of EIPs:

There are three primary categories of EIPs:

EIP Type Description
Standards Track Changes affecting the Ethereum protocol or application-level standards (e.g., ERCs like ERC-20)
Meta Processes, guidelines, and decision-making (e.g., EIP-1 that defines how EIPs work)
Informational General guidelines or information not meant to be enforced

üß† Key Points:

  • EIPs start as drafts, get discussed in the community (GitHub, Ethereum Magicians, etc.), and then may be accepted and finalized.
  • Accepted EIPs that define application-level standards (like tokens) become ERCs (Ethereum Request for Comments).

2. ERC-20 and ERC-721 (and Other ERC Standards)

üîπ What Is an ERC?

ERC stands for Ethereum Request for Comments. These are application-level standards based on EIPs and define how smart contracts should behave in specific use cases ‚Äî most commonly, tokens.

üè¶ ERC-20: Fungible Token Standard

ERC-20 defines the standard interface for fungible tokens ‚Äî tokens that are identical and interchangeable (like USDC, DAI, or your voting power in a DAO).

üì¶ Real-World Example:

  • 1 USDC = 1 USDC (no uniqueness)
  • Ideal for currencies, governance tokens, DeFi assets

üîß Core ERC-20 Functions:

Function Purpose
totalSupply() Returns the total number of tokens in existence
balanceOf(address) Returns the balance of a specific address
transfer(to, amount) Transfers tokens from the caller to another address
approve(spender, amount) Authorizes another address (a dApp, for example) to spend tokens
allowance(owner, spender) Shows how much a spender is allowed to spend from an owner's balance
transferFrom(from, to, amount) Allows a spender to transfer tokens on behalf of an owner (with approval)

üß† ERC-20 Events:

Event Purpose
Transfer Emitted when tokens are moved
Approval Emitted when an allowance is set

üñºÔ∏è ERC-721: Non-Fungible Token (NFT) Standard

ERC-721 is the standard for non-fungible tokens ‚Äî tokens that are unique and not interchangeable. Each token has its own ID and metadata.

üì¶ Real-World Example:

  • NFT art, in-game items, virtual land
  • Token ID #124 ‚â† Token ID #125

üîß Core ERC-721 Functions:

Function Purpose
balanceOf(owner) Returns how many NFTs the owner owns
ownerOf(tokenId) Returns who owns a specific NFT
safeTransferFrom(from, to, tokenId) Safely transfers an NFT (checks recipient can receive it)
transferFrom(from, to, tokenId) Transfers an NFT without safety check
approve(to, tokenId) Gives another address permission to transfer a specific NFT
getApproved(tokenId) Returns the approved address for a token
setApprovalForAll(operator, approved) Gives or removes permission for an operator to manage all NFTs
isApprovedForAll(owner, operator) Checks if operator is approved to manage all of owner's NFTs

üß† ERC-721 Events:

Event Purpose
Transfer Emitted when ownership of an NFT changes
Approval Emitted when an address is approved to manage a specific NFT
ApprovalForAll Emitted when an operator is approved for all tokens

üìö Other Important ERC Standards:

ERC Use Case Notes
ERC-1155 Multi-token standard (supports both fungible and non-fungible tokens in one contract) Gas-efficient for games
ERC-4626 Tokenized vault standard for DeFi protocols Used in yield farming, e.g., Yearn Vaults
ERC-777 Advanced fungible token with hooks Backward-compatible with ERC-20, less commonly used
ERC-2981 Royalty standard for NFTs Specifies how creators get royalties on resales

üß† How These Standards Help Developers:

  • Interoperability: All wallets, dApps, and block explorers can interact with tokens using the same interface.
  • Security: Audited and community-approved structure reduces bugs.
  • Ecosystem Support: OpenZeppelin provides libraries that implement these standards securely.

1. ERC-20 Token Code Example:


// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MyToken is ERC20 {
    constructor(uint256 initialSupply) ERC20("MyToken", "MTK") {
        _mint(msg.sender, initialSupply * 10 ** decimals());
    }
}

üß† Key Components:

Line Description
ERC20("MyToken", "MTK") Token name and symbol
_mint(msg.sender, amount) Creates tokens for deployer
decimals() Defaults to 18 (can be overridden)

2. Implementing an ERC-721 (NFT) Token with OpenZeppelin


// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/utils/Counters.sol";

contract MyNFT is ERC721 {
    using Counters for Counters.Counter;
    Counters.Counter private _tokenIds;

    constructor() ERC721("MyNFT", "MNFT") {}

    function mintNFT(address recipient) public returns (uint256) {
        _tokenIds.increment(); // Auto-increment ID
        uint256 newItemId = _tokenIds.current();
        _mint(recipient, newItemId); // Mint NFT to recipient
        return newItemId;
    }
}

üß† Key Components:

Element Purpose
ERC721("MyNFT", "MNFT") Token name and symbol
_tokenIds Tracks current NFT ID
_mint() Mints NFT with unique ID to user

üñºÔ∏è For metadata (image, description, etc.), you‚Äôd typically extend this using ERC721URIStorage.

4. Best Practices

  • ‚úÖ Use OpenZeppelin‚Äôs battle-tested contracts
  • ‚úÖ Inherit from ERC20Burnable, Ownable, Pausable, or AccessControl as needed
  • ‚úÖ For NFTs, use ERC721Enumerable for tracking token supply and ERC721URIStorage for metadata
  • ‚úÖ Avoid over-minting or unaudited mint logic
  • ‚úÖ Use constructor-based minting or gated mint functions (e.g., onlyOwner)
